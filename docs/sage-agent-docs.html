1. Executive Summary

Prime Directive: The core experience — voice interaction, memory retrieval, simple task execution, and basic code running — must function entirely on-device with no network connection. Cloud services extend capability; they are never required for the app to be useful.

SAGE-AGENT is a production-grade, privacy-first mobile AI companion built for iOS and Android. It combines Siri-grade OS integration, Replit-grade code execution, and Perplexity-grade synthesis — with a hard architectural guarantee: the device is the agent, not a thin terminal to a cloud.

The project resolves a fundamental tension in modern AI: to get a highly personalized assistant, users currently must stream their entire life to a megacorporation's servers. SAGE-AGENT breaks this Faustian bargain by placing the full intelligence stack on-device, using cloud resources strictly as a stateless, key-isolated proxy for tasks that genuinely exceed local capability.

2. Problem Statement

2.1 The Cloud Dependency Crisis

Current AI assistants are architecturally glass screens rendering cloud intelligence. This creates four compounding business problems:

📡 Connectivity Dependency

Every powerful AI tool becomes a paperweight during outages, flights, remote work, or poor cellular coverage. Users cannot trust AI for critical tasks.

🔐 Surveillance Tradeoff

Personalization requires streaming private data to centralized servers — contacts, calendar, voice, location. The user has no cryptographic guarantee their digital twin is not being mined or sold.

💸 Marginal Cost Spiral

Cloud inference at scale is expensive. Every token of every user's every query is billed to the operator, making sustained, high-frequency personal AI economically unsustainable.

🧩 App Silo Lock-in

Users manage intent across dozens of apps. There is no unified intelligent layer that understands context across calendars, files, contacts, and tasks without siloing that context in a vendor's cloud.

2.2 Market Opportunity

Modern smartphones ship with 8–16GB RAM, dedicated Neural Processing Units, and GPU acceleration (Metal on iOS, Vulkan on Android). The hardware has outpaced the software paradigm. A 2B-parameter GGUF model runs in under 2GB of RAM at 30+ tokens/second on an iPhone 15 Pro — yet no production assistant exploits this.

3. Business Vision & Goals

3.1 Vision: The Airplane Mode Singularity

The ultimate business achievement is the moment a user at 30,000 feet, completely disconnected, can ask their device to:

  • Search through a year of local meeting notes via on-device semantic retrieval
  • Write and execute a JavaScript script to parse a downloaded CSV file
  • Generate a functioning HTML UI for a new project and render it on-screen

This is the Airplane Mode Singularity — the proof that continuous, high-level cognition does not require a tether to a server farm.

3.2 Strategic Business Goals

Goal 1 — Absolute Data Sovereignty

All memory retrieval, routing decisions, and OS integrations execute locally. The backend is a stateless proxy; it never stores, searches, ranks, or embeds user memory. Users have a cryptographic guarantee that their digital twin cannot be mined, sold, or breached from a centralized database.

Goal 2 — The Death of the App Paradigm

SAGE-AGENT replaces the app store model with intent-driven dynamic software. Instead of downloading a mortgage calculator, users ask SAGE to write, run, and save one. The user becomes a commander of ephemeral software rather than a consumer of pre-packaged apps.

Goal 3 — Democratized Thick-Client AI

Prove that a $500 smartphone is capable of hosting a sovereign software factory and research lab. By running frontier-quality intelligence locally and escalating only the most complex edge cases to cloud, the marginal cost of intelligence approaches zero.

Goal 4 — Return Ownership to the User

Transform the smartphone from a terminal that connects you to an AI into the AI itself. The device does not borrow intelligence from a server — it hosts it.

6. Business Rules (Constitutional Constraints)

These are binding business rules that cannot be overridden by any implementation decision. Violations constitute a product defect, not a trade-off.

C1
Device Owns the Intelligence Loop

The mobile device drives all reasoning and tool orchestration. The cloud server executes exactly one inference cycle per request and returns. No server-side planning, looping, or state between calls. Business reason: prevents cloud lock-in and preserves offline capability.

C2
Routing Decisions Are Never Made Server-Side

The on-device SageRouter selects model and target (local vs cloud). The server applies an allowlist check only — it never overrides the device's routing decision. Business reason: routing decisions expose user context; they must not leave the device.

C3
The Backend Is a Stateless Proxy

POST /api/sage/infer validates, forwards to the upstream, streams back, and returns. No state persists between calls. Business reason: a stateless backend cannot be subpoenaed for user conversation history.

C4
Two-Domain Tool Registry Is Authoritative

Every tool is categorized as either mobile (on-device, offline) or cloud (requires connectivity). A mobile tool never executes on the server; a cloud tool never executes without connectivity. Business reason: prevents accidental data exfiltration via tool arguments.

C5
Memory Is On-Device and Never Synced

The sqlite-vec memory store is local-only. The backend receives memories as opaque prompt text and never stores, searches, ranks, or embeds them. Deletion is real — no shadow copy. Business reason: this is the foundation of the data sovereignty promise.

C6
No Features on Deprecated Paths

The legacy /api/sage/agent endpoint, the vm sandbox, and native tool mocks are in maintenance mode only. No new capabilities are built against them. Business reason: prevents architectural debt from undermining the clean v3 contract.

1. Product Overview & The Four Pillars

SAGE-AGENT is a mobile AI companion structured around four mutually reinforcing pillars. Each pillar has a defined offline path before any online path (the Prime Directive).

Pillar A — Voice & OS Interface

Offline: Wake word (Porcupine), STT (Whisper.cpp), TTS (Piper) all run locally via llama.rn. Deep OS access (Contacts, Calendar, Reminders, Files) available offline via native bridges.

Online: Same pipeline; routing can escalate to cloud model for complex responses.

Target: <500ms end-to-end voice I/O latency.

Pillar B — Hybrid Code Sandbox

Offline: QuickJS isolated context for JS/TS execution and data manipulation. Sandboxed WebView for HTML/JS prototype rendering. Both comply with App Store Guideline 2.5.2.

Online: E2B Firecracker microVM for Python and heavy compute (cloud domain tool; ToolDomainRouter returns OFFLINE envelope when disconnected).

Pillar C — Adaptive Search & Synthesis

Offline: sqlite-vec semantic search over user's personal notes and device history. Must return top-5 results in <100ms at 100k vectors.

Online: Tavily web search + Jina Reader page fetch, orchestrated by the mobile ReActLoop using cloud domain tools.

Pillar D — SageCore

The on-device orchestration brain. Three independently testable components:

  • SageRouter — evaluates 5 signals, selects model/target
  • ReActLoop — drives all reasoning and tool orchestration
  • ToolDomainRouter — two-domain dispatch for all 13 tools

1. Architectural Paradigm: Thick Client / Thin Server

Core Architectural Decision: SAGE-AGENT inverts the dominant "thin client / thick server" AI paradigm. The mobile device is the orchestration brain; the Express backend is a stateless proxy — "Hands" that reach upstream resources the device cannot access directly.

The trust boundary between device and server is absolute:

Device Responsibilities (everything that touches user data)

  • All reasoning and tool orchestration (the ReActLoop)
  • Routing decisions (the SageRouter)
  • Memory storage, retrieval, and relevance scoring (sqlite-vec)
  • Local inference (llama.cpp GGUF)
  • Voice pipeline (Porcupine → Whisper.cpp → Piper)
  • Mobile domain tools (all 9: QuickJS, WebView, Contacts, Calendar, Reminders, Files, Memory)
  • Capability manifest and feature gating

Server Responsibilities (stateless, keyless proxy)

  • One inference cycle per request (validate → allowlist check → forward → stream → return)
  • Upstream API key isolation (OpenRouter / Azure AI Foundry / Tavily / E2B keys never reach the device)
  • Cloud domain tool runtime (web_search, fetch_webpage, execute_python, deep_research)
  • SSE streaming the Backend v3 contract back to the device

3. Core Architectural Patterns

3.1 The ReActLoop — Mobile-Owned Orchestration

The ReActLoop is the central intelligence loop. It runs entirely on-device; the server never knows how many iterations have occurred. Each cycle:

  1. Calls SageRouter → gets {model, target, rationale}
  2. If target=local: runs on-device via llama.cpp binding (LocalInferenceTarget)
  3. If target=cloud: calls POST /api/sage/infer and consumes the SSE stream (CloudInferenceTarget)
  4. On tool_call event: delegates to ToolDomainRouter
  5. Appends tool_result to messages and loops
  6. On text event: yields to UI / TTS
  7. On retryable error: descends the graceful degradation hierarchy

3.2 The SageRouter — 5-Signal Evaluation

Evaluates exactly 5 signals before every inference request. Never contacts the network during evaluation.

SignalSourceValuesHard Override
Network QualityOS network monitor (NetInfo)offline / poor / fair / goodoffline → always local
Device Power StateOS battery API + thermal guardcritical / low / normal / chargingcritical → always local efficient
Task ComplexityOn-device heuristic classifiersimple / moderate / complex—
Privacy ContextApp state + user preference storestandard / private / sensitivesensitive → always local
User PreferenceEncrypted MMKV settings storeauto / prefer_local / prefer_cloudprefer_local → always local

3.3 The Two-Domain Tool Registry

Every tool has an authoritative domain stamp. The ToolDomainRouter dispatches by the registry's domain — not the domain the model places on a tool call — so a model can never trick the device into sending mobile tool arguments to the server.

3.4 The SSE Contract

The Backend v3 streams exactly 5 event types. The device's SSE parser and the server's SSE serializer are generated from the same @sage/sse-contract source, making wire compatibility a compile-time guarantee.

6. Security Design

6.1 API Key Isolation

No upstream API key (OpenRouter, Tavily, E2B, Azure AI Foundry) ever touches the device. The mobile client authenticates only to SAGE Backend v3 using a session token (stored in Keychain/Keystore). The backend holds all upstream credentials and applies only an allowlist check before proxying requests — it never overrides the device's model selection.

6.2 Encryption at Rest

StoreContentsEncryption
WatermelonDBConversations, skills, agent stateSQLCipher (AES-256); key from Keychain/Keystore
sqlite-vecMemory vectors + textSame SQLCipher database and key
MMKVSettings, KV cache, routing preferencesMMKV encryptionKey (AES); key from Keychain/Keystore

6.3 Code Execution Sandboxing (App Store 2.5.2)

Generated JS code is data, not app code. It executes inside a fresh QuickJS runtime+context per run with:

  • No bridge to native modules, the React tree, file system, or network
  • Enforced memory limit (64MB default), stack limit (512KB), and interrupt handler (1s timeout)
  • Only a capturing console.log injected — no other host globals
  • Context torn down after every execution (no shared state between runs)

HTML prototypes render in a locked-down WebView with all native bridges disabled, file:// access denied, and no postMessage channel to native.

7. Data Flow: Local & Cloud Inference Cycles

7.1 Local Target Cycle (Offline / Hard Override)

User voice/text
  → ReActLoop.run(messages, memories)
  → SageRouter.route({signals, capability})
  → { model: 'gemma-4-2b', target: 'local' }     ← hard override (offline/critical/sensitive/prefer_local)
  → LocalInferenceTarget.run(messages)
  → llama.cpp GGUF on-device inference (Metal/Vulkan)
  → stream: text tokens | <tool_call> blocks
  → ToolDomainRouter.dispatch(call)               ← mobile domain only
  → MobileToolHandler (QuickJS / OS APIs / sqlite-vec)
  → tool_result appended → next iteration
  → TTS (Piper, on-device)

7.2 Cloud Target Cycle

User voice/text
  → ReActLoop.run(messages, memories)
  → MemoryManager.recall(transcript, 5)           ← on-device, never sent raw
  → SageRouter.route({signals, capability})
  → { model: 'anthropic/claude-sonnet-4', target: 'cloud' }
  → CloudInferenceTarget.fetch(POST /api/sage/infer, {
      model, messages, memories: [...opaque fragments...]
    })
  → SAGE Backend v3:
      validate → allowlist-check model → inject memories verbatim
      → upstream (OpenRouter / Azure AI Foundry)
      → SSE stream: chunk | tool_call | done | error | heartbeat
  → Device decodes SSE stream
  → On tool_call: ToolDomainRouter.dispatch(call)
      - cloud domain → POST /api/sage/tools/* (Tavily / E2B)
      - mobile domain → on-device (never server-routed)
      - offline + cloud → OFFLINE ToolResult
  → tool_result appended → next iteration
  → TTS (Piper, on-device)

4. Algorithm Specifications

4.1 SageRouter Routing Algorithm

Evaluated in strict precedence order. The output is RoutingDecision {model, target, rationale}:

function route(signals, capability):

  // 1. HARD OVERRIDES — always local, no exceptions
  if signals.network === 'offline'        → local, model=chooseLocal(complexity, capability, forceEfficient=false)
  if signals.power === 'critical'         → local, model=DEFAULT_LOCAL (2B, efficient tier)
  if signals.privacy === 'sensitive'      → local, model=chooseLocal(complexity, capability)
  if signals.preference === 'prefer_local'→ local, model=chooseLocal(complexity, capability)

  // 2. EXPLICIT CLOUD PREFERENCE — skip soft guidance
  if signals.preference === 'prefer_cloud'
    → cloud, model = complexity==='complex' ? cloud.capable : cloud.efficient

  // 3. SOFT GUIDANCE — lean local when marginal
  if signals.network === 'poor' AND signals.complexity === 'simple'
    → local, model=DEFAULT_LOCAL
  if signals.power === 'low' AND signals.complexity !== 'complex'
    → local, model=DEFAULT_LOCAL

  // 4. DEFAULT — cloud, tier by complexity
  → cloud, model = complexity==='complex' ? cloud.capable : cloud.efficient

function chooseLocal(complexity, capability, forceEfficient=false):
  if forceEfficient: return LOCAL_MODELS.default  // 2B always for critical battery
  if complexity==='complex' AND has9B(capability): return LOCAL_MODELS.capable  // 9B
  return LOCAL_MODELS.default  // 2B

4.2 Graceful Degradation Hierarchy

function buildDescent(decision):
  if decision.target === 'cloud':
    attempts = [
      {cloud, decision.model},          // router's choice first (may be capable or efficient)
      {cloud, cloud.efficient},          // step down to efficient cloud (if not already there)
      {local, bestLocalModel()},         // cross to local (never escalates back up)
      {local, LOCAL_MODELS.default}      // floor: 2B efficient model
    ]
  else: // local decision = offline/privacy/prefer_local
    attempts = [
      {local, decision.model},           // chosen model
      {local, LOCAL_MODELS.default}      // floor: 2B
    ]
    // Local decisions NEVER escalate to cloud.

On each attempt:
  - If first event is 'error' AND retryable AND hasNextAttempt → onDegrade(from, to) → descend
  - If first event is 'error' AND NOT retryable → surface immediately (no degradation)
  - If first event is any other type → commit to this attempt, stream remainder

4.3 Complexity Classifier (Signal 3)

Deterministic heuristic; no model, no network; same input always yields same output:

function scoreComplexity(text):
  score = 0
  if wordCount(text) > 60:   score += 2
  elif wordCount(text) > 25: score += 1
  if hasCodeCues(text):      score += 2   // ```, =>, function, class, import, const, def
  if isResearchy(text):      score += 2   // research, analyze, compare, synthesize, cite...
  if researchVerbCount >= 2: score += 1   // stacking multiple research verbs
  if isMultiStep(text):      score += 1   // "and then", "after that", numbered lists
  if isDeepQuestion(text):   score += 1   // why, how, explain, derive, prove

  if score >= 4: return 'complex'
  if score >= 2: return 'moderate'
  return 'simple'

4.4 Memory Relevance Scoring (for capacity pruning)

// Used by MemoryManager.prune() to evict lowest-relevance records beyond capacity.
// Higher score = more worth keeping.
function relevanceScore(record, now):
  ageMs = max(0, now - record.lastAccessedAt)
  recency = 0.5 ^ (ageMs / HALF_LIFE)      // HALF_LIFE default = 30 days
                                             // → 1 when just accessed, → 0 as it ages
  frequency = log2(record.accessCount + 1)  // log scale; avoids runaway inflation
  return recency + 0.25 * frequency

4.5 Memory Recall Procedure

function recall(query, topK=5):
  queryEmbedding = l2normalize(embedder.embed(query))
  raw = store.search(queryEmbedding, topK * 2)   // over-fetch to allow expiry filtering
  hits = []
  for hit in raw:
    if isExpired(hit.record, now):
      store.delete(hit.record.id)               // lazy expiry on read path
      continue
    hit.record.lastAccessedAt = now
    hit.record.accessCount += 1
    store.upsert(hit.record)                    // bump access stats
    hits.append(hit)
    if len(hits) >= topK: break
  return hits
